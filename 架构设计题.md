## 模块一：Context / Performance（上下文与性能优化）

### 2.一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

我们不能简单粗暴地"截断最老的对话"——那样 Agent 会像失忆一样，你说"刚才那个事"它完全不记得。最佳实践是**分层记忆衰减**，分三层来处理：

**第一层：短期记忆（无损保留）**
最近 10-20 轮的对话，**原封不动保留**。这是为了保证当前沟通的流畅性——"那个"、"刚才说的"这类指代能正常消解。

**第二层：中期记忆（摘要压缩）**
20 轮以前的对话，用一个便宜的轻量模型（比如 8B 参数的端侧模型）在后台异步生成"滚动摘要"。用几百字概括之前的关键事件、决策和讨论脉络，然后把原始 Token 替换掉。相当于你读完一本书后写的读书笔记——保留了精华，丢了废话。

**第三层：长期记忆（结构化抽取）**
压缩的过程中不只是生成摘要，还要提取出结构化的信息——用户的偏好、职业身份、提到过的人名。这些东西存到 KV 数据库或图数据库里，作为 System Prompt 的外挂变量。这样核心人设永远不会丢。

## 模块二：Memory（记忆机制）

###   1.和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

简单的向量检索在这里存在一定的问题——**跨时态的语义漂移**。用户的表达方式会变，当时的"那个需求"和现在的"那个需求"向量可能对不上。

更靠谱的做法是 **混合检索 + 时间衰减**，分三步走：

**第一步：意图识别 + 时间解析**
先判断用户是不是在翻历史记录，然后提取出时间约束（"半个月前" = 具体日期范围）。

**第二步：混合检索，双路召回**
同时跑两种检索，互相兜底：

- **密集检索**：把问题做语义向量化（Embedding），搜语义相近的。
- **稀疏检索**：提取关键词做 BM25 匹配，补上向量在专有名词上的短板。

**第三步：时间衰减重排**
这个问题的关键在于——用户明确说了"半个月前"。所以不能只靠语义打分，必须在向量库里直接加时间过滤条件（Time Filter），只搜那个时间窗口内的内容，然后再按相关性排序。

## 模块三：Task（任务管理）

### 1.对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

三个方案，按实现复杂度从低到高排列：

**方案 A：Global System Prompt 锚定（最简单）**
我们在每次调用 LLM 时，在 System Prompt 最顶端塞两行：
```
Global Objective: [全局目标]
Current Step: [当前执行到哪一步]
```
> 优点：实现简单，对模型注意力牵引效果明显。

> 缺点：步骤一多，模型还是会忘。毕竟长上下文推理是模型本身的硬伤。

**方案 B：State Machine / DAG 检查点**
把长任务拆成一个有向无环图（DAG）。每个节点只做一件原子事，输出标准化的状态字典。下一个节点只看全局状态和当前任务指令，**不读之前乱七八糟的思考过程**。

> 优点：彻底隔离了无关 Token 的干扰，容错率极高，还支持断点续传——崩了能从中间节点重来。

> 缺点：工程量大，需要预先设计好任务路由和状态 Schema。

**方案 C：Reflection & Self-Correction（反思机制）**
每执行 3-5 步，强制调用一个独立的"审计 Agent"，专门比对当前产出和初始目标。不一致就报错回滚。

> 优点：大模型的幻觉往往具有“自我确证”的倾向。独立的审计节点能在 3-5 步这个容错窗口期内及时拉停，避免后续成百上千个 Token 的无用功。

> 缺点：审计 Agent 本质上也是 LLM，同样无法摆脱幻觉和过度对齐。它可能会变得过于严苛，对其实已经正确的产出“挑刺”，导致系统在某个节点不断重试、互相推诿，最终触发死循环并耗尽 Token 额度。

## 模块四：Tool / Session Runtime（工具与会话运行时）

### 2.如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

**1：用户在 Busy 时发来新消息**

- **普通消息**：塞进**消息缓冲队列**，等当前生成完成后再处理。不打断，不乱序。
- **"“停止/打断指令"**：立即发送 Cancel Token，强杀当前 LLM 的生成进程，然后处理新的指令。

**2：异步工具的结果回来了**

工具完成回调到达时，**不要去打断正在进行的 LLM 生成**。把结果静默写入数据库的 Context 流。等当前生成结束、状态机变为 Idle 时，Runtime 发现 Context 里有未处理的 Tool Response，自动触发下一轮推理。

## 模块五：Agent Runtime 架构对比

### 1.Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

**Claude Code 风格：纯文本解析（XML / Markdown / Bash）**

优点：
- 极度符合语言模型的"天性"——模型本来就是文字游戏的专家
- 代码、解释、工具调用混在一起输出时特别自然
- 容错率高——XML 标签没闭合也能猜出意图
- Coding 场景下效果极佳

缺点：
- 得自己写正则或 AST 解析器去提 `<tool_call>` 标签
- 没有统一的强类型校验
- 标签闭合错了可能整段解析失败

**OpenAI 风格：强类型 JSON Schema**

优点：
- 标准化程度极高，生态成熟
- API 层直接做 JSON 校验，拿到的就是结构化数据
- 契合传统软件工程的强类型习惯

缺点：
- 长文本和结构化数据混在一起输出时，流式解析体验较差
- 复杂的嵌套 Schema 很耗 Token
- 模型容易在缺逗号、括号这种语法细节上翻车
